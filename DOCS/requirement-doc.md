# 信息收集系统 - 需求文档 v1.0

## 一、项目概述

本系统旨在构建一个**自动化信息收集与采集平台**，分为三大模块：

| 模块 | 名称 | 说明 |
|------|------|------|
| 模块一 | 信息收集模块 | 定义收集目标与数据来源 |
| 模块二 | 采集程序模块 | 通用采集引擎 + 规则适配层，独立部署运行 |
| 模块三 | 定制化处理模块 | 预留，依赖模块二的标准输出 |

---

## 二、模块一：信息收集模块

**输入**：业务场景 + 具体需收集的信息字段（含必填与可扩展字段）

**数据来源**（初始 + 自主扩展）：

| 来源类型 | 示例 | 策略 |
|----------|------|------|
| 社交媒体 | 小红书、抖音 | 优先 API，兜底浏览器渲染 |
| 公众号 | 微信公众号 | 抓取文章列表/正文页 |
| 网页/博客 | 任意公开文章页 | 直接抓取 HTML |
| 公共数据平台 | 天气/财报/宏观数据 | 结构化 API 或页面解析 |
| 其他 | 待发现 | 随需求扩展 |

**负责人**：用户定义收集目标，Agent 扩展数据来源。

---

## 三、模块二：采集程序模块（核心）

### 3.1 通用采集引擎

- **核心定位**：与 AI 解耦，纯规则驱动，可独立部署在任意云平台
- **工作方式**：加载 YAML 规则配置 → 执行采集 → 格式化输出 → 状态上报
- **触发方式**：在 YAML 规则中配置触发事件（定时任务 / 手动触发 / Webhook 等）
- **运行方式**：常驻进程/定时任务，全自动执行，无需人工介入

### 3.2 规则设计（YAML）

每个来源对应一个 YAML 规则文件，规则实现标准化接口。规则文件由人工手动编写维护，随项目部署一起管理。

**YAML 规则示例**：

```yaml
name: "小红书笔记采集规则"
version: "1.0.0"

source:
  platform: "xiaohongshu"
  url: "https://www.xiaohongshu.com/..."
  trigger:
    type: "cron"              # cron | manual | webhook
    schedule: "0 */6 * * *"  # 每6小时，仅 trigger.type=cron 时有效
  auth:
    type: "cookie"           # cookie | api_key | oauth | none
    credential: "${COOKIE}"  # 环境变量引用（也支持直接写死值）
  render:
    enabled: true            # 是否启用浏览器渲染
    wait_for: 3000           # 渲染等待毫秒数

fields:
  - name: "title"
    selector: "meta[property='og:title']"
    required: true
  - name: "content"
    selector: ".note-content"
    required: true
  - name: "author"
    selector: ".author-name"
    required: true
  - name: "likes"
    selector: ".like-count"
    type: "number"

crawl:
  incremental: true          # 增量采集，默认开启
  dedup_key: "note_id"       # 增量去重字段

output:
  format: "json"
  path: "./output/"          # 输出路径

alert:
  on_failure: true
  retry_times: 3
```

**认证方式支持**：

| 方式 | 适用场景 |
|------|----------|
| `none` | 公开页面 |
| `cookie` | 需要登录态的页面（如小红书、公众号） |
| `api_key` | 有 API Key 的数据平台 |
| `oauth` | 第三方授权场景 |

凭证支持直接写入配置文件，也支持环境变量引用。

### 3.3 渲染策略

- **优先策略**：优先寻找公开 API 接口，减少资源消耗
- **兜底策略**：无 API 时启用无头浏览器渲染（Headless Chrome / Playwright）
- **可配置项**：渲染等待时间、滚动深度、SPA 路由等待等

**资源评估**：

- 推荐配置：**4核8G**（含无头浏览器渲染场景）
- 低于 4G 内存时，无头浏览器渲染可能出现 OOM 或极慢，不建议在低资源环境使用

### 3.4 状态监控（Web Dashboard）

- **技术栈**：Vue + 纯 HTML/CSS（轻量看板，不做复杂交互）
- **功能定位**：只读展示，不包含对 YAML 规则文件的增删改查
- **展示内容**：
  - 规则列表 + 启用/禁用状态
  - 采集任务状态（待执行/执行中/成功/失败）
  - 增量采集进度（已采集 / 总数）
  - 最近 N 条采集元信息
  - 错误日志与重试记录

### 3.5 预警通知

- **触发条件**：连续采集失败达到阈值（如 3 次）
- **通知方式**：日志记录（当前阶段）
- **日志规范**：
  - **成功日志**：记录来源、采集数量、耗时、时间戳（**不含原始数据内容**）
  - **失败日志**：记录来源、错误原因、重试次数、时间戳
  - **日志文件**：结构化写入指定日志目录，支持后续对接告警渠道

### 3.6 输出格式

- **统一输出**：JSON 结构化数据
- **输出路径**：本地文件目录（可配置路径）
- **标准化**：所有来源输出遵循统一 JSON Schema，模块三可直接消费

## 四、全局去重机制

### 4.1 设计背景

一条采集需求可能对应多个来源（小红书 + 公众号 + 知乎 + 微博……），每个来源各自增量采集，但需要跨来源做全局去重，避免同一内容被多个来源重复采集。

### 4.2 存储方案

SQLite 文件（`dedup.db`），无需额外部署服务，随程序一起运行。

### 4.3 表结构

```sql
CREATE TABLE dedup (
    id          TEXT PRIMARY KEY,      -- 全局唯一ID，格式：平台_原始ID
    requirement TEXT NOT NULL,          -- 采集需求标识，如 "ai_model_news"
    platform    TEXT NOT NULL,          -- 来源平台，如 xiaohongshu / wechat / zhihu
    url         TEXT,                   -- 原URL，可用于后续处理或回查
    collected_at TEXT NOT NULL,         -- 采集时间 ISO格式
    raw_id      TEXT                   -- 来源平台的原始ID，如小红书的 note_id
);

CREATE INDEX idx_requirement ON dedup(requirement);
CREATE INDEX idx_platform ON dedup(platform);
```

### 4.4 全局唯一 ID 生成规则

统一格式：`平台_原始ID`

| 平台 | 示例 ID |
|------|---------|
| 小红书 | `xhs_note_abc123` |
| 微信公众号 | `gzh_xyz789` |
| 知乎 | `zh_art_456` |
| 微博 | `wb_post_111` |

### 4.5 采集流程中的增量处理

```
加载规则 → 读取全局去重表 →
新内容与去重表比对（按 requirement 范围）→
只采不在表里的内容 → 采集完成后写入去重表 → 输出 JSON
```

**说明**：已采集内容不会重复采集；如果来源内容被删除/下架，不在当前版本处理范围内（可后续扩展）。

## 六、后期扩展需求（记录于此）

以下需求暂不在第一版范围内，记录留待后续版本：

1. **Dashboard 规则管理** — 支持在 Web UI 上对 YAML 规则文件进行增删改查
2. **在线规则编辑器** — Dashboard 内置 YAML 在线编写/测试功能
3. **服务热重启** — 规则变更后无需手动重启进程，服务自动 reload
4. **告警通知渠道扩展** — 对接飞书/钉钉/Email 等即时通知渠道
5. **数据入库** — 采集结果先入库 SQLite/数据库，再查询输出，支持历史数据检索
6. **规则市场/模板** — 预置常见来源的规则模板，一键启用

## 七、技术栈汇总

| 组件 | 技术选型 | 说明 |
|------|----------|------|
| 规则格式 | YAML | 手动管理，有编写说明文档 |
| 核心语言 | Python | 采集引擎主体 |
| 浏览器渲染 | Playwright | 无 API 来源兜底 |
| Web 框架 | Vue + 纯 HTML/CSS | 轻量看板，不做复杂交互 |
| 输出格式 | JSON | 结构化，统一 Schema |
| 日志 | 结构化日志文件 | 全量记录成功/失败元信息 |
| 去重存储 | SQLite | 全局去重表，所有采集需求共享 |
